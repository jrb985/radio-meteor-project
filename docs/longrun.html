What it looks like operationally

  • Run the CLI, NOT the GUI, for long unattended sessions:
    • python meteor_detector.py (prints pings, writes CSV + snapshots)
    • Memory is bounded: preallocated IQ ring + a BYTE-bounded snapshot queue.
    • It will run flat for days; a 5-min heartbeat prints pings/reconnects/RSS.
  • The GUI is for WATCHING, not overnight logging:
    • it keeps every thumbnail image + growing lists in memory -> balloons over hundreds/thousands of events.
  • Morning workflow:
    • python session_report.py -> pings-per-hour chart + summary browse snapshots/ (classified meteor / aircraft / interference).

Limitation 1: USB glitch ends the run -- FIXED (v1.3)

  • Over 12 h there is a real chance of:
    • a USB hiccup, a libusb read error, or Windows power-managing the dongle.
  • WAS: if sdr.read_samples() threw, the capture loop exited and the rest of the night was lost -- a single transient failure = no data until morning.
  • NOW: reconnect.py (ReconnectingReader) wraps every read.
    • On error: close -> exponential backoff (1 s, capped at 30 s) -> reopen -> re-apply freq/gain -> resume. Default: never give up.
    • An in-progress event is discarded and the IQ ring cleared (sample-index continuity breaks across the gap); the noise-floor baseline is kept.
    • Reconnects are counted and shown in the 5-minute heartbeat.
  • Config: reconnect_enabled / _backoff_start_s / _backoff_max_s / _give_up_after.
Slide 5 of 12

Limitation 2: Snapshot rendering blocks capture -- FIXED (v1.3/v1.4)

  • WAS: snapshots rendered SYNCHRONOUSLY inside the capture loop. matplotlib savefig takes ~150-300 ms per event -- and while rendering, no samples are read, so the RTL-SDR USB buffer can overflow and DROP samples. During a burst that cascades into continuous dropouts.
  • NOW: snapshot.py SnapshotWorker renders on a BACKGROUND thread.
    • The capture thread only extracts the IQ window and classifies it (cheap), then hands the job off -- submit is non-blocking.
    • The queue is bounded by BYTES (v1.4): a job holds a whole event window, 4-25 MB, so a job-COUNT bound bounded nothing.
  • Backpressure = the governor: over the cap, the SNAPSHOT is dropped and counted -- the EVENT is still classified and written to the CSV. The counts (the science) never degrade under load.

Limitation 3: Seattle air traffic -> disk flood

Near SeaTac, 104.3 forward-scatter catches many aircraft reflections; each currently saves a PNG.

ScenarioEvents / 12 hPNGs (~200 KB ea)If raw IQ on (~12 MB ea)
Quiet~50~10 MB~0.6 GB
Busy (aircraft / E-skip)~1,500~300 MB~18 GB
  • Keep snapshot_save_raw_iq = False (it already is) -- raw IQ is the disk killer.
  • Even PNG-only, a busy night is a few hundred MB and a LOT of images to triage.
  • NOW GOVERNED: snapshot_skip_aircraft skips exactly these, and the byte-bounded queue drops renders (never events) under burst. On the Pi, .npz is ~1-2 MB/event.

Limitation 4: Windows will try to interrupt you

  • System sleep / hibernation: kills capture -> must be disabled.
  • USB selective suspend: can suspend the dongle -> disable it.
  • Windows Update auto-restart: can reboot mid-run -> pause updates / active hours.
  • Fine to leave on: display sleep, screensaver.
  • All handled in the pre-flight checklist (next-next slide).
Slide 8 of 12

Limitations 5 & 6

  • 5. Thermal drift -- mostly a NON-issue here:
    • the tuner drifts as it warms, but we detect over the WHOLE +-90 kHz FM channel, so a few kHz of drift is irrelevant.
    • (It would matter for a narrowband CW beacon; it does not for us.)
  • 6. Analysis burden:
    • a busy night's snapshots are dominated by aircraft.
    • The auto-classifier + per-class session report exist precisely to make triage manageable -- but review is still real work.

Pre-flight checklist (for when you do it)

  • 1. powercfg /change standby-timeout-ac 0 (no system sleep).
  • 2. Disable USB selective suspend (dongle + USB Root Hubs, Power Mgmt tab).
  • 3. Pause Windows Update for the run window.
  • 4. Confirm free disk (budget ~1 GB to be safe).
  • 5. Verify nothing else holds the dongle.
  • 6. Launch with a log file: python meteor_detector.py *> run_YYYYMMDD.log 7. Ideally time it to a shower (Perseids Aug 12-13); rates rise toward dawn.
  • 8. Spot-check after ~15 min that pings / CSV / snapshots are flowing, then leave it.
Slide 10 of 12

Robustness upgrades -- BUILT (v1.3 / v1.4)

  • These are what make an overnight run trustworthy. All now in the code:
    • Auto-reconnect: retry on USB/read error (close -> backoff -> reopen -> resume). reconnect.py. [limitation 1]
    • Async snapshots: background worker + bounded queue, so rendering never stalls the reader. snapshot.py SnapshotWorker. [limitation 2]
    • Snapshot governor: skip 'aircraft' events and/or drop renders over a BYTE cap -- bounds disk, CPU and memory; events still logged. [limitation 3]
    • Heartbeat: every 5 min the CLI prints uptime, pings, reconnects, RSS, and snapshots saved/dropped -- health confirmable at a glance.
  • Verified HEADLESSLY (synthetic sources). A real-hardware overnight run is still the outstanding validation step.

Summary & recommendation

  • The code-side blockers are GONE: auto-reconnect (a glitch no longer ends the night) and async snapshots (bursts no longer drop samples) are BUILT, plus a governor that bounds disk/CPU/memory. A 10-12 h CLI run is ready to attempt.
  • What is left is OPERATIONAL, and it is on you, not the code:
    • Windows power/update pre-flight (sleep OFF, USB selective suspend OFF, Updates paused, ~1 GB free disk).
    • Log to a file, and spot-check at ~15 min that pings/CSV/snapshots flow.
    • Neither v1.3 nor v1.4 has yet run against real hardware overnight -- treat the first night as a shakedown as much as an observation.
  • Keep snapshot_save_raw_iq=False; leave the snapshot governor on.
  • Best payoff: run it on a shower night (Perseids Aug 12-13) and compare pings-per-hour to the expected rise toward dawn.
