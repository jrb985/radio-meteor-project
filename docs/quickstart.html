Quickstart

Two ways to run the station — pick the one that matches where the Pi is. Everything below is typed on the Pi (over SSH, or on its own terminal).

Path A

📶 At home — on the network

The Pi is powered and on your Wi-Fi (even if it's physically outside). Runs 24/7 and self-publishes a status page you can watch from anywhere.

  1. Get onto the Pi and bring it up to date. Move into the project folder, update the OS packages, then pull the latest detector code. 
    cd ~/radio-meteor-project
sudo apt-get update && sudo apt-get upgrade -y   # keep Raspberry Pi OS current
git pull                                          # latest detector code

    A big upgrade can occasionally ask for a reboot — if so, sudo reboot, then cd ~/radio-meteor-project again and continue.

  2. Make sure field mode isn't armed (Path B and Path A can't both run — they share the dongle). 
    systemctl is-enabled meteor-field 2>/dev/null || echo "(not armed — good)"
# if it prints "enabled":
sudo systemctl disable meteor-field
  3. Start the detector. The program that does the work is meteor_detector.py; the meteor-tracker service runs it for you and restarts it on boot or after a crash. 
    sudo systemctl enable --now meteor-tracker   # start now + on every boot
sudo systemctl restart meteor-tracker        # if you just pulled new code

    Want to run it by hand in the foreground instead (to watch it live or test)? Stop the service first so they don't share the dongle:

    sudo systemctl stop meteor-tracker
METEOR_PROFILE=pi .venv/bin/python meteor_detector.py   # Ctrl+C to stop
  4. Confirm it's alive and detecting. Watch for the Listening: line, then events and a [heartbeat] within 5 minutes. 
    systemctl status meteor-tracker --no-pager
tail -f meteor.log
  5. The status page publishes itself every hour (a cron job renders and pushes it). Confirm it's scheduled, or push an update right now: 
    crontab -l                            # the hourly job should be listed
bash deploy/pi/publish_status.sh      # publish immediately
    Live status https://jrb985.github.io/radio-meteor-project/status.html

    Allow ~1–2 min after a push for GitHub Pages to rebuild. Prefer a real-time view on the same network? Run python deploy/pi/status_server.py and open http://<pi-ip>:8080 — no internet needed for that one.

Stopping it

Run sudo shutdown -h now, wait for the green activity LED to go dark, then unplug. Never pull power while it's still blinking — that can corrupt the SD card.

Path B

🎒 In the field — unattended, no internet

Plug it in, it observes for a set number of hours, then powers itself off. No keyboard, no network, no operator. Do the setup once at home; after that it's power-on-and-walk-away.

Before you leave — do this once, at home

  1. Bring the Pi up to date and install the field unit. 
    cd ~/radio-meteor-project
sudo apt-get update && sudo apt-get upgrade -y
git pull
bash deploy/pi/install_field.sh
  2. Set how long each run observes (default is 8 hours — skip this step to keep 8). 
    sudo systemctl edit meteor-field
# in the editor, add exactly these two lines under a [Service] header:
#   [Service]
#   Environment=FIELD_HOURS=8
  3. Arm it. This swaps out the always-on service so the two never fight for the dongle. It takes effect at the next boot. 
    sudo systemctl disable --now meteor-tracker
sudo systemctl enable meteor-field

In the field

Just power on the Pi. That's the whole operation. It will:

  • observe for FIELD_HOURS, logging every event to meteor_events.csv;
  • stop the detector cleanly (finalizing the exposure log);
  • and power itself off. Safe to unplug once the green activity LED goes dark.

Every power-on repeats the cycle — so to run again, just power-cycle it.

When you get home

  1. Pull the data. Copy meteor_events.csv and snapshots/ off (over the network or by pulling the SD). What happened each run is in field_boot.log.
  2. Switch back to home / 24-7 mode. 
    sudo systemctl disable meteor-field
sudo systemctl enable --now meteor-tracker

The status page is offline in the field

With no internet, the hourly publisher can't push — that's expected. Your science is safe on the SD card regardless; the live page just resumes once the Pi is back on the network in Path A.

Reference

Which mode am I in?

One command tells you which service is set to run at boot — exactly one should be enabled:

systemctl is-enabled meteor-tracker meteor-field
# meteor-tracker enabled  + meteor-field disabled  = Path A (home, 24/7)
# meteor-tracker disabled + meteor-field enabled   = Path B (field box)

Full detail on monitoring, station health, and safe shutdown is in the operator's guide.
