Operations · Deployment

Running the station

The detector runs headless on a Raspberry Pi 3B+. This page is the operator's guide: where to watch it, how to read its health, how to stop it without corrupting the SD card, and how to build a plug-in-and-walk-away field box that observes a fixed session and powers itself off.

01 · Monitoring

Where it actually logs

The service sends the detector's output to a file, not the systemd journal.

The systemd unit sets StandardOutput=append:meteor.log, so every event, snapshot line, and the 5-minute [heartbeat] goes to meteor.log — not to journalctl. Running journalctl -u meteor-tracker shows only systemd's own lifecycle chatter (“Started…”); it looking quiet is normal and does not mean the detector is idle.

# the detector's real output: events, snapshots, heartbeat
tail -f meteor.log

# the science record — one row per detected ping
tail -f meteor_events.csv

# is the process alive / restart-looping? (systemd's view)
systemctl status meteor-tracker

# live dashboard on the local network
python deploy/pi/status_server.py        # http://<pi-ip>:8080

Give it five minutes and you should see a [heartbeat] line — that's the confirmation the capture loop is genuinely running.

02 · Health

Station health at a glance

The live page and the published page both carry a color-coded health grid.

Equipment health is read straight from /proc, /sys and vcgencmd (no extra dependencies):

MetricWhy it matters on a bare 3B+
CPU / SoC tempSustained DSP load can thermally throttle the Pi (~80–85 °C). This is the one temperature that actually exists to read.
Power / throttlevcgencmd get_throttled — under-voltage trips here first, then shows up as USB read errors. The single most useful field-health signal.
Load averageAgainst the 4 cores — is the detector keeping up with the sample stream.
RAM free / detector RSSThe number that matters in 1 GB. The pi profile holds RSS near 100 MB.
Disk freeSnapshots and logs accumulate; a full SD card ends a run silently.
Uptime / reconnectsSystem uptime, and how many USB glitches the in-process reconnect recovered.

No dongle thermometer

RTL-SDR dongles (RTL2832U + R820T) expose no temperature sensor — there is nothing to read over USB. The under-voltage state plus the reconnect count stand in as the dongle's health signal, since that is exactly how a hot or under-powered dongle fails.

03 · Stopping safely

Never pull power from a running Pi

An abrupt cut is the one move that can actually break the deployment.

Yanking power mid-write can corrupt the SD-card filesystem (occasionally badly enough that it won't boot) and skips the detector's clean shutdown, which finalizes the CSV and the observing-exposure log — the record that keeps the diurnal analysis honest. Already-logged pings are usually safe (the CSV flushes after every event); the SD and the exposure finalize are not.

The always-safe habit

Run sudo shutdown -h now, wait for the green activity LED to go dark (only the red power LED steady), then unplug. Never pull power while the activity LED is still blinking.

Under the hood, the detector now treats SIGTERM like Ctrl+C: a systemctl stop, a timeout, or an OS shutdown all run its clean-shutdown path and finalize the logs. (Previously only Ctrl+C did — a plain service stop killed it mid-write.)

04 · Attended field run

Observe N hours, then power off

One command, one sudo prompt up front, then walk away.

field_run.sh schedules the power-off, runs the detector for the window, stops it cleanly, and lets the OS halt itself — so it is safe to unplug when you return:

bash deploy/pi/field_run.sh 8      # observe 8 h, then power off (default 8)

You enter your sudo password once (it schedules shutdown -h +N); the run itself is unattended. It stops the always-on service first so they don't contend for the dongle, runs the detector under timeout (SIGTERM → clean finalize), and the OS does a proper sync + unmount before halting. Changed your mind mid-run? sudo shutdown -c.

05 · No-touch field box

Power on → run → power off, unattended

For a box you plug in and leave — no SSH, no keyboard, no network needed.

Install the boot unit, then arm it (arming swaps the always-on service out so they never share the dongle):

bash deploy/pi/install_field.sh                # installs meteor-field.service (unarmed)

# arm it — takes effect at the NEXT boot:
sudo systemctl disable --now meteor-tracker
sudo systemctl enable meteor-field

The cycle is then power on → observe 8 h → clean shutdown → safe to unplug once the green LED goes dark, and every power-on repeats it. How it stays safe:

  • The unit runs as root so it can poweroff with no password…
  • …but field_boot.sh immediately drops to your user via runuser for the detector, so meteor_events.csv and snapshots/ stay owned by you — no root-owned-file surprises if you switch modes.
  • The detector runs under timeout; the SIGTERM at the deadline triggers the same clean finalize, then the Pi powers off.
  • It always powers off — even if the dongle wasn't seen at boot (a field box has no operator to rescue it). The reason lands in field_boot.log; pull the SD to read it.

Change the window: sudo systemctl edit meteor-field → add [Service] then Environment=FIELD_HOURS=6.
Back to 24/7: sudo systemctl disable meteor-field then sudo systemctl enable --now meteor-tracker.

Field mode is offline

The hourly status-page publisher does not run in field mode (it needs the network, and a field box likely has none). The exposure log and meteor_events.csv are the record — copy them (or the whole SD) when the box comes home.
