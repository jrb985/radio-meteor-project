The data pipeline

RF in -> ping out, through the modules

RF in (antenna)→RTL-SDR IQ samples→Band-power + baseline→Detect ping (thresh/hyst)→Classify (inline)→CSV row + snapshot job
  • IQ read continuously at 1.024 Msps; each ~4 ms block -> FFT power in the +-90 kHz channel.
  • EventDetector: EMA noise-floor baseline + threshold/hysteresis -> a Ping (with sample indices).
  • On a ping: the IQ window is pulled from the ring buffer and classify.py labels it INLINE (cheap, so the label is ready for the CSV row). The snapshot is handed to a background SnapshotWorker (v1.3).
  • KEY INVARIANT: under load the SNAPSHOT is dropped, never the EVENT -- every ping still reaches the CSV, so the counts (the science) never degrade under memory or CPU pressure.

Modules: who does what

  • config.py -- all tunable params (frozen dataclass) + PROFILES (e.g. 'pi').
    • rtl_sdr_env.py -- registers the bundled librtlsdr DLL dir (imported FIRST).
  • Detection core (in meteor_detector.py):
    • BandPowerAnalyzer (FFT band power) -> EventDetector (EMA baseline + threshold/hysteresis) -> CsvSink (writes rows).
  • snapshot.py -- IQRingBuffer (preallocated circular) holds recent IQ;
    • extract_window pulls the event; SnapshotWorker writes it off-thread.
  • dsp.py -- chunked float32 spectrogram + band_stats. Shared by classify and the renderers; keeps the per-event FFT transient at a few MB, not ~200 MB (v1.4).
  • classify.py -- labels each ping from its IQ window (runs INLINE, so it must be cheap).
  • reconnect.py -- ReconnectingReader: survives USB/read errors (v1.3).
  • capture_engine.py -- runs the SAME core on a background thread, publishing to a queue (injectable source -> headless testable).
  • gui_app.py -- Tkinter UI drains that queue. CLI and GUI SHARE the core modules.

The operational workflow

The steps you actually follow

1. Setup→2. Verify→3. Choose frequency→4. Observe→5. Analyze
  • 1. Setup: bind WinUSB (Zadig), edit config.py. 2. Verify: check_device.py, spectrograph.py.
  • 3. Choose frequency: fm_scan.py / fm_vacancy.py (already set to 104.3 MHz).
  • 4. Observe: meteor_detector.py (CLI, long runs) or gui_app.py (GUI).
  • 5. Analyze: session_report.py (a session) and diurnal.py (many days).

Stages 1-2: Setup & Verify

  • Setup (once):
    • Bind WinUSB to the dongle via tools/rtl-sdr/zadig.exe (Interface 0).
    • Edit config.py (frequency, gain, thresholds, snapshot options).
  • Verify & explore:
    • check_device.py -- opens the dongle, captures IQ, prints tuner/stats.
    • fm_scan.py -- strongest carriers (reception sanity check).
    • fm_vacancy.py -- ranks channels -> find a VACANT one for scatter.
    • spectrograph.py -- one-shot image; live_spectrograph.py -- live waterfall.
    • monitor.py N -- timed headless capture -> power + waterfall PNGs.
Slide 6 of 12

Stage 3: Observe -- CLI vs GUI (same engine)

  • meteor_detector.py (CLI):
    • Best for LONG / unattended runs -- bounded memory, logs to a file.
    • Prints each ping; writes meteor_events.csv + snapshots/.
  • gui_app.py (GUI):
    • Best for WATCHING -- Live tab (waterfall + side gallery) and Gallery tab (big image viewer). Live freq/gain/threshold, ping counter, Report button.
    • Not for overnight (thumbnail/list memory grows).
  • Both run the identical BandPowerAnalyzer / EventDetector / snapshot / classify code -- the GUI just wraps it in capture_engine.py on a thread.
Slide 7 of 12

Stage 4: Analyze (science tools)

  • session_report.py -- one session: pings-per-hour stacked by class + summary (meteors/hr, span, per-class counts). Also reachable from the GUI 'Report' button.
  • diurnal.py -- MANY days: folds meteor-only counts onto local hour-of-day, exposure-normalized with Poisson error bars (the diurnal-curve first project).
  • Inputs: meteor_events.csv (+ the snapshots/ images for visual triage).
  • Filter matters: analyze meteor-CLASS events; aircraft/sporadic-E contaminate rates.
  • Reference reading + the science rationale: references/ and Science_and_Publishability.pptx.
Slide 8 of 12

Profiles: one codebase, two machines (v1.4)

  • Memory is dominated by SNAPSHOTS, not detection. At 1.024 Msps one second of IQ is 8.2 MB, and a queued snapshot job holds a whole event window (4-25 MB).
  • config.py PROFILES['pi'] -- select with METEOR_PROFILE=pi or --profile pi:
    • snapshot_mode='npz' -- the Pi does NOT render. matplotlib is never imported (~90 MB of RSS) and no multi-second render runs on a slow ARM core.
    • snapshot_queue_max_BYTES -- bounding the queue by job COUNT bounds nothing.
    • snapshot_max_window_s / _max_rows -- cap any one job; shrink the ring.
    • snapshot_skip_aircraft -- near SeaTac these dominate the burst.
  • Measured (tools/mem_soak.py, synthetic event flood):
    • pi profile ~104 MB peak RSS | desktop defaults ~450-490 MB The Pi captures; your PC renders: render_snapshots.py turns .npz -> PNG.

Data artifacts & how they connect

  • meteor_events.csv -- one row per ping: start_utc, duration_s, peak_snr_db, peak_power_db, classification. (Old 4-col files auto-rotate to .old.)
  • snapshots/ping_<UTC>_<class>.png -- one zoomed spectrogram per ping (snapshot_mode ='png', the Windows default); the class is in the filename.
  • snapshots/ping_<UTC>_<class>.npz -- the Pi's compact float16 spectrogram (~1-2 MB);
    • render to PNG on a PC with render_snapshots.py. Same image either way.
  • sessions.csv (v1.5) -- the EXPOSURE log: append-only on/beat/off rows saying when the receiver was actually detecting (excludes warm-up + reconnect gaps).
    • Every RATE divides by this. Without it, exposure has to be guessed -- and a night-only schedule then fabricates a textbook diurnal curve. See uptime.py.
  • Diagnostics: monitor_power.png / monitor_waterfall.png, spectrograph.png.
  • Analysis outputs: session_report.png, diurnal.png.
  • Everything downstream keys off meteor_events.csv + sessions.csv + snapshots/.

Summary

  • Data path: RF -> IQ -> band-power/baseline -> ping -> classify+snapshot -> CSV/snapshots -> analysis.
  • Human path: setup -> verify -> choose frequency -> observe (CLI or GUI) -> analyze.
  • CLI and GUI share one detection core; the GUI adds a threaded engine + tabs.
  • One codebase, two machines: config.py PROFILES['pi'] fits the same core in 1 GB.
  • Stable contract: meteor_events.csv + snapshots/ feed all analysis. Under load the snapshot is dropped, never the event -- the counts always survive.
